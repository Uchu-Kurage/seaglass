## この作品について

実況天気と連動する、静かなビーチコーミングのゲーム。
地図には載っていない「あなただけの海岸」で、石・貝殻・シーグラスをさがす。
本質は **発見の喜び** と **秘密の海岸らしい親密さ**。急かさず、静かで、詩的であること。

公開URL: <https://uchu-kurage.github.io/seaglass/>

## 作品の核となる判断基準（コードより優先すること）

以下は機能や見た目を判断するときの「憲法」です。効率や機能の豊富さより、これらを優先します。

- **静けさを壊さない。** 通知・演出・UIは控えめに。にぎやかさや達成の煽り、派手なアニメーションは避ける。潮騒のような穏やかさを保つ。
- **詩的な抽象を、具体的な正確さより優先する。** 海岸名は必ず架空・叙情的にする（例:「硝子の浜」「北風の岬」）。実在の地名は出さない。座標は天気の多様性のためだけに内部で使い、画面には出さない。
- **現実の時間のリズムを尊重する。** 漂着物はおよそ1時間に1個のペースで、1日の上限内で打ち上がる。浜を離れている間も溜まる。この「時々のぞきに来る」ゆっくりした感覚が作品の背骨。壊さない。
- **毎日ひらきたくなる仕掛けは、自然のリズムそのものに宿す。** 連続ログインやストリーク、焦らせる通知のような圧力型は入れない。潮・天気・月・収集の歩みといった、世界の側のゆらぎで静かに誘う。ゲージ・数値・達成演出は出さず、フィードバックは極控えめに保つ。
- **一つ一つが、その人だけのものであること。** 石やシーグラスの形は seed から生成し、二度と同じ形にしない。拾ったものは「棚」に一点ずつ、いつ・どの浜で拾ったかと共に記録される。この個体性を薄めない。
- **自分だけの場所である感覚。** 浜には自分で名前をつけられる。命名の儀式のような、私的で大切な体験を保つ。

判断に迷う変更（特に見た目・トーンに関わるもの）は、勝手に決めず、PR の説明にトレードオフを書いて相談する形にしてください。

## 技術構成

- **フレームワーク:** React 18 + Vite 5
- **描画:** Canvas 2D（`src/App.jsx` 内で完結）。浜・波・空・漂着物はすべて手描画。
- **天気:** [Open-Meteo](https://open-meteo.com/)（APIキー不要）。取得失敗時は「体験モード」に自動フォールバック。過去天気（`past_days`）は「時化のあとの漂着」判定に使い、当日1回だけ取得してキャッシュする。
- **潮・月:** 潮位は月齢のみから算出する半日周潮の近似（航海用途ではない）。月齢計算は一箇所に置き、他機能からも再利用する。
- **フォント:** Shippori Mincho（Google Fonts）
- **デプロイ:** GitHub Actions で `main` への push を自動ビルド・公開（`.github/workflows/deploy.yml`）。

コードは現状ほぼ `src/App.jsx` の1ファイルに集約されている。大きくなってきたら分割してよいが、
分割のための分割はしない。まとまりとして自然な単位（描画・データ・ストレージ・画面）で切ること。

## 海のリズム（実装済みの主な系）

現実の月と天気に同期した「今日だけの海」で、静かな再訪動機をつくる系。詳細仕様は `docs/feature-sea-rhythm.md`。

- **潮の満ち引き。** 干潮で浜が広がり、潮間帯の個体が現れる。干潮時刻は毎日約50分ずれ、浜ごとに異なる。潮位は時刻由来の連続値で、アイテムの状態とは分離する。
- **時化のあとの漂着。** 前日〜未明の実天気が荒れた浜に、翌朝 流木やレア色ガラスが打ち上がる。翌日まで残り、通常の1日上限とは別枠。
- **稀に流れ着く小瓶。** 確率ではなく収集マイルストーンで出現する。通常色＝日常の層、レア色＝核心の層の二系統で、両層を集めきると物語の全容が解ける。
- **共有基盤。** 日付シード決定性 `dateSeed`、0:00 の日境界統一、月齢計算の再利用。

## 確定した設計判断（再検討しない）

以下は対話で決着済み。仕様変更の明示指示がない限り、蒸し返さず前提として扱う。

- **日境界は 0:00。** 1日上限リセット・漂着・小瓶の「今日」判定をすべてこれに揃える。
- **潮間帯の同時露出は最大3個。**
- **漂着は翌日まで残存**（翌々日の朝に潮が引き取る）。**1日上限とは別枠。**
- **流木は収集物**（棚・カタログに計上する）。
- **小瓶は二系統マイルストーン**（通常色/レア色）。同日1本まで、同時到達は核心を優先。両系統完走で全容が解ける。
- **フィードバックは極控えめ。** ゲージ・数値・通知・潮の時刻表示を出さない。浜が広がること自体が合図。

## 物語の正典：小瓶

小瓶のメッセージには物語が伏せてある。原稿と設計は **`docs/bottle-narrative.md` が正典**。以下を守る。

- **テキストは正典のものをそのまま使う。** 文言の創作・改変・外部作品からの引用をしない。UI へは同ドキュメントの `bottleMessages` データを用いる。
- 「あなた」は **恋人**。不在は **死別だが、明示語で書かない**（気配で示す）。
- 鍵の色は **赤**（この渚でいちばん稀な色）。核心の層のトリガーで、機能②の時化限定色に含める。
- 断片は **日常10・核心5**。通し読みは日常が核心を額装し、**赤の一片（解放）で閉じる**。
- 一人称は **わたし／常体**。**暗に示す**を貫き、言い切らない。
- **実在地名・固有名を出さない。「あなた」の名も伏せる。** プレイヤーが名づけた浜の名は本文で使わない。

## 触るときに壊してはいけない約束

- **セーブは必ず「両対応」を保つ。** `src/App.jsx` 冒頭の `storage` オブジェクトは、通常ブラウザでは `localStorage`、
Artifact プレビュー環境では `window.storage` を使うようにしてある。
この二重対応を絶対に外さないこと。`localStorage` 直書きや `window.storage` 直書きに戻さない。
セーブのキーは `seaglass-` 接頭辞で統一（`seaglass-collection` / `seaglass-finds` / `seaglass-beach-names` / `seaglass-shore-<beachId>`）。**海のリズム系で足す保存キーも同じ接頭辞に統一する。**

- **旧セーブデータとの後方互換を保つ。** コレクションは数値（旧）とオブジェクト（新）の両形式を読めるようにしてある
（`entryCount` / `entrySeed` ヘルパー）。データ構造を変えるときも、
既存プレイヤーのセーブが壊れない移行を用意すること。**新フィールドは未定義でも壊れないよう、読み込み時にデフォルト補完する。**

- **`base` パスの自動判定を壊さない。** `vite.config.js` の `base` は `GITHUB_REPOSITORY` から自動で決まる。
リポジトリ名が変わっても動くこの仕組みを残すこと。

- **打ち上げペースの定数は一箇所に。** `WASH_INTERVAL` / `DAILY_CAP` / `MAX_ON_SHORE` / `FIRST_VISIT` は `src/App.jsx` 上部にまとまっている。ペース調整はここだけで完結させる。潮・漂着・小瓶のしきい値定数も、同じように一箇所へまとめる。

- **形も位置も、決してリセットしない。** 描画の依存は `beach.id` に留め、`beach` オブジェクトの再生成を描画に巻き込まない（過去に「1個拾うと他が全部リセット」する回帰を起こした箇所）。潮で水没した潮間帯の個体や、未採取の漂着も state から消さず、位置は相対座標(0..1)とシードで永続化する。

- **「今日の状態」は乱数ではなく決定論で。** 潮・漂着・小瓶の当日の姿は `dateSeed(date, beachId, salt)` から導き、リロードや再描画で揺れないこと。乱数の引き直しをしない。

- **キャンバスの初期化タイミングに注意。** 命名画面から浜へ遷移した直後はキャンバス高さが 0 になりうる。
漂着物の初期配置は高さが確定してから行う（`seedFromShore` / `S.seeded` の仕組み）。
ここを触るときは、遷移直後に漂着物が空の上へ出るバグを再発させないこと。

- **一括置換で抽象化レイヤを壊さない。** `sed` などの全置換は使わず、対象を限定した編集を行う（過去に `storage` 抽象化の条件分岐を全置換で壊した）。複雑な多箇所編集のあとは括弧・テンプレートリテラルの対応を確認する。

## 開発の進め方

- 変更は必ずブランチを切り、PR を出す。`main` に直接コミットしない。
- PR の説明には「何を・なぜ変えたか」と、上の判断基準に照らしたトレードオフを一言添える。
- 動作確認:
  

```
npm install
npm run dev      # http://localhost:5173/
npm run build    # 本番ビルドが通ることを確認
```

- 見た目・トーンに関わる変更は、可能ならスクリーンショットや短い説明を PR に添える。
