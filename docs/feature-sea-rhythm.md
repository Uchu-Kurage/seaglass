# 機能仕様：海のリズム（潮・時化のあとの漂着・小瓶）

> 対象リポジトリ: `seaglass`
> 想定配置: `docs/feature-sea-rhythm.md`
> 実装環境: Claude Code（本ドキュメントを起点に PR を作成する）
> 関連: `CLAUDE.md`（創作の憲章）, `docs/feature-return-greeting.md`（留守のあいだの挨拶）

---

## 0. 目的とトーン

「毎日ひらきたくなる」動機づけを、**プレッシャー型（連続ログイン・ストリーク・焦らせ通知）ではなく、自然のリズムそのもの**で作る。

- 現実の月と天気に同期した"今日だけの海"を用意し、来訪の理由を世界の側から差し出す。
- どうぶつの森の時間帯要素、Pokémon Sleep の「昨夜どうだったか」を毎朝確かめる感覚に近い。ただし数値目標やゲージは前面に出さない。
- 既存の創作価値を厳守する：**スロータイム**（急がせない）／**内密な発見**（実在地名を出さない）／**シードされた個体性**（形は決してリセットしない）。

本仕様は次の3機能を扱う。互いに軽く連結するが、**それぞれ独立した PR として実装可能**。

1. 潮の満ち引き（Tides）
2. 時化のあとの漂着（Storm wash-ups）
3. 稀に流れ着く小瓶（Message bottles）

---

## 1. 共有基盤（先に入れる）

3機能が共通で使う土台。**PR #1 として最初に用意する**ことを推奨。

### 1.1 日付シード決定性 `dateSeed(dateKey, beachId, salt)`

再描画やリロードで**結果が揺れてはならない**（過去の「beach 再生成で全アイテムがリセット」「形は決してリセットしない」の教訓と同じ原則）。潮・漂着・小瓶の"今日の状態"は乱数ではなく、**日付とビーチから決定論的に導出**する。

- 既存のシード付き乱数生成関数を再利用する。無ければ mulberry32 / xmur3 相当を1つ用意。
- `dateKey`（例 `"2026-07-18"`）と `beachId` と用途別 `salt`（`"storm"` `"bottle"` 等）を文字列連結してハッシュ → シード → RNG。
- 同じ日・同じビーチ・同じ用途なら、何度呼んでも同一系列を返すこと。

```
seededRandom(hashString(`${dateKey}:${beachId}:${salt}`)) → () => number in [0,1)
```

### 1.2 日境界（1日の区切り）の統一

- **日境界は 0:00（ローカル深夜0時）に統一する（確定）。** 潮以外の"今日"判定（漂着・小瓶）と、既存の「1日上限24個」リセットは、すべてこの 0:00 境界に揃える。
- 二重定義しない。境界判定は 0:00 基準の共通関数を1つ用意し、全機能から参照する。
- ⚠️ 既存の24個リセットが 0:00 以外を使っている場合は、0:00 に合わせて調整する。

### 1.3 保存の二重互換

- 新規保存キーもすべて `localStorage` / `window.storage` フォールバックの抽象化レイヤ経由で読み書きする。
- ⚠️ `sed -i` 等の全置換で抽象化レイヤ自体を壊した過去があるため、変更は対象を限定した `str_replace` で行う。

### 1.4 世界観ガード

- 潮・天気・座標に由来する情報を UI に出す際、**実在の緯度経度・地名を絶対に露出させない**。座標は天気/潮の計算にのみ内部利用。
- 数値（波高◯m 等）を生表示しない。表現は情景語に寄せる（「凪」「時化」など）。

## 2. 機能①：潮の満ち引き（Tides）

### 2.1 体験

- 一日のなかで汀線（波打ち際）が上下し、**干潮のときだけ浜が広がる**。
- 干潮時にのみ姿を見せる場所（潮間帯）があり、普段は海の下にある個体をそこで拾える。
- 干潮の時刻は**毎日およそ50分ずつ後ろへずれていく**。「今日の引きは何時ごろだろう」で来訪動機になる。
- ビーチごとに潮の時刻が少し異なる（内密な個体性）。

### 2.2 潮位モデル（感覚重視の近似・航海用途ではない）

APIを増やさず、**月齢のみから決定論的に**算出する。半日周潮（M2, 周期約12.42h）を基本に、月齢で大潮/小潮を揺らす。

```
T_TIDE       = 12.42            // 時間。半日周潮の周期
SYNODIC      = 29.53            // 朔望月
lunarAge     = 新月からの経過日数（既存の月齢計算があれば再利用）

// ビーチ差：経度から時差様のラグを与える（座標は内部利用のみ）
//   longitude が使えない場合は beachId ハッシュから定数ラグを与える
beachLagH    = -longitude / 15   // フォールバック: hash(beachId) % 12

// 毎日約50分ずれる潮時計（T_TIDE≠12h のため自然にドリフトする）
phase        = frac((hoursSinceEpoch - beachLagH) / T_TIDE)   // 0..1
baseHeight   = cos(2π * phase)                                 // +1 満潮, −1 干潮

// 大潮/小潮：新月・満月で大きく、上下弦で小さく（月2回の大潮）
springFactor = abs(cos(2π * lunarAge / SYNODIC))               // 0..1
amplitude    = 0.55 + 0.45 * springFactor                      // 0.55..1.0

tideLevel    = amplitude * baseHeight                          // −1..+1
```

- `tideLevel` は **−1（最干）〜+1（最満）**。
- 再計算頻度は **毎フレーム不要**。1分ごと（または既存のスポーン tick）に更新し、canvas へ渡す。
- ⚠️ 月齢を出す既存関数があれば必ず再利用（満月機能を将来入れる際の一貫性のため）。

### 2.3 描画（Canvas）

- `tideLevel` から汀線 Y を算出：`waterlineY = lerp(Y_lowTideLine, Y_highTideLine, (tideLevel+1)/2)`。
- 満潮：汀線が上がり、潮間帯を覆う。干潮：汀線が下がり、潮間帯が露出。
- 汀線付近に**濡れ砂の帯**と淡い反射を描くと情景が締まる（既存の波・天候・時刻ライティングと同レイヤで扱う）。
- 波アニメーションの基準線を `waterlineY` に追従させる。

### 2.4 スポーン統合（最重要・リセット厳禁）

スポーン領域を2帯に分ける。

| 帯 | 位置 | 露出条件 |
|---|---|---|
| 常設帯 | 高潮線より上 | 常に露出（**既存スポーンはすべてここ**） |
| 潮間帯 | 高潮線〜低潮線 | `tideLevel` が閾値以下（例 `< -0.2`）のとき露出 |

- **潮間帯の個体も、生成時に位置とシードを確定して state に保持する。** 満潮で水没しても state から消さず、その間は「採取不可・水面下として非表示（または半透明）」で描くだけ。露出したら再び採取可能に戻る。
- これにより**形・位置は決してリセットされない**。潮位は時刻由来の連続値なので、汀線アニメはアイテム state から**完全に分離**する（`beach` オブジェクトの再生成を描画に絡めない。過去バグと同型に注意：依存は `beach.id`）。
- 潮間帯の個体は「引いている間の限定的な出会い」。**上限（1日24個）には含めない別枠（確定）**。"特別な時間"として扱う。
- **潮間帯の同時露出上限は3個（確定）。** 引いている間に同時に見える潮間帯の個体は最大3個まで。

### 2.5 フィードバック（控えめに）

- **プロミネンスは極控えめ（確定）。** 明示ゲージ・数値・通知は置かない。**浜が広がること自体が唯一の合図**。
- 潮時刻の表示や満/干のアイコンも出さない。プレイヤーが自分で気づくに任せる（内密な発見）。漂着も同様に、演出で知らせず、来たときに見つける。

### 2.6 受け入れ基準

- [ ] リロード・再描画で汀線が飛ばず、既存アイテムの位置・形が変わらない。
- [ ] 干潮時に潮間帯が露出し、そこに固有シードの個体が現れる／満潮時は覆われるが state からは消えない。
- [ ] 干潮時刻が翌日およそ+50分ずれる。ビーチごとに時刻が異なる。
- [ ] 大潮（新月・満月付近）で振幅が大きく、小潮で小さい。

## 3. 機能②：時化のあとの漂着（Storm wash-ups）

### 3.1 体験

- 前日〜未明にそのビーチの**現実の天気が荒れていた**ら、翌朝、汀線に**流木や珍しい色のガラス**が打ち上がっている。
- 「昨日は北風の岬が荒れていたな」と、現実の天気がゲームへの誘いになる。
- 「留守のあいだに、流木が三つ 打ち上がっていました」のように**帰還挨拶へ自然接続**できる。

### 3.2 データ取得（Open-Meteo）

- 既存の Open-Meteo 利用に**過去分**を足す。`forecast` API の `past_days`（過去数日）で前日の hourly を取得。
- 判定指標（いずれか / 重み付け）：`wind_gusts_10m` 最大、`precipitation` 合計、（海に面する座標なら Marine API の `wave_height` 最大）。
- ⚠️ 座標がマリン非対応の場合は wave を諦め、**風＋降水にフォールバック**。
- **キャッシュ必須**：`(dateKey, beachId)` 単位で当日1回だけ取得し保存。APIを叩き続けない。

```
STORM_GUST_KT   = 34         // これ以上の最大瞬間風速で時化寄り（要調整）
STORM_PRECIP_MM = 10         // 前日降水合計の目安
STORM_WAVE_M    = 2.0        // マリン利用時（要調整）

stormIntensity ∈ [0,1]  // 上記を正規化・合成した荒れ具合
isStorm = stormIntensity >= 0.4   // 閾値は要調整
```

### 3.3 生成ロジック（決定論・朝の一度きり）

- 漂着は**その日・そのビーチにつき一度だけ**確定する朝のイベント。`dateSeed(dateKey, beachId, "storm")` の RNG を使い、リロードで揺れないこと。
- 個数は intensity に比例するが**小さく上限**（例 `count = clamp(round(stormIntensity * 4), 1, 5)`）。スロータイムを壊すジャックポットにしない。
- 種類：
  - **流木**：新規の**収集物カテゴリ（確定）**。大きめ。図書室の種カタログ／棚にも計上する。
  - **珍しい色のガラス**：既存シーグラスの色プールに**時化限定のレア色**を追加し、通常スポーンには出さない。形は既存のシード生成をそのまま使い、一つずつ固有にする。
- 配置は**汀線の帯（潮間帯／打ち上げライン）**。→ 機能①の汀線と連動させると自然。

### 3.4 寿命と上限

- 上限（1日24個）とは**別枠（確定）**。特別イベントのため。ただし 3.3 の個数上限で抑える。
- 寿命：**翌日まで残す（確定）。** 打ち上がった日と翌日いっぱいは汀線に残り、それ以降（翌々日の朝の再計算）で潮が引き取る。急かさず、しかし放置しすぎないリズム。

### 3.5 保存スキーマ（例）

「翌日まで残す」ため、**個体ごとに打ち上がった日 `appearedDate` を持ち**、ビーチ配下の `items` は日をまたいで**蓄積**する（毎日上書きしない）。

```json
"washups": {
  "<beachId>": {
    "lastCheckedDate": "2026-07-18",
    "items": [
      { "seed": 1837465, "type": "driftwood", "colorId": null, "x": 0.42, "y": 0.71,
        "appearedDate": "2026-07-17", "collected": false },
      { "seed": 9182003, "type": "seaglass", "colorId": "storm-amber", "x": 0.55, "y": 0.68,
        "appearedDate": "2026-07-18", "collected": true }
    ]
  }
}
```
- `x,y` は 0..1 の相対座標で保存（キャンバスサイズ非依存・リセット防止）。
- **朝の再計算（0:00 境界後の初回起動）で2段階処理する**：
  1. **清掃**：`today - appearedDate > 1日` の**未採取**個体を除去（潮が引き取る）。採取済みは図書室に入っているので `items` から外してよい。
  2. **生成**：当日ぶんの `lastCheckedDate` が未処理なら、§3.2–3.3 に従い当日の漂着を判定・生成し、`appearedDate = today` を付けて `items` に**追加**する。
- これにより、前日ぶん（未採取）と当日ぶんが**汀線に共存**しうる。

### 3.6 帰還挨拶との接続

- `docs/feature-return-greeting.md` の「留守のあいだに、◯つ 届いていました」に、**時化由来のコピー**を追加できるようフックを用意（例：荒天フラグが立った日は文面を差し替え）。実際のコピーは Ryoga が最終調整。

### 3.7 受け入れ基準

- [ ] 前日が荒天のビーチだけ翌朝に漂着が出る／穏やかなら出ない。
- [ ] 同日中はリロードしても個数・種類・位置が不変。
- [ ] 未採取の漂着は翌日いっぱい残り、翌々日の朝に消える（前日ぶんと当日ぶんが共存しうる）。
- [ ] 流木は収集物として図書室に計上される。
- [ ] 過去天気の取得は当日1回にキャッシュされ、API を連打しない。
- [ ] レア色ガラスは時化経由でしか入手できない。

## 4. 機能③：稀に流れ着く小瓶（Message bottles）

### 4.1 体験

- シーグラスを集めていると、そのお礼のように小瓶がひとつ流れ着く。中には**短い言葉の断片**。
- 断片には**二つの層**がある：
  - **日常の層**（通常色を集めると届く）…主人公の日々・気配・待つ時間。物語を直接は語らない。
  - **核心の層**（レア色を集めると届く）…物語の芯に触れる断片。なぜ主人公が硝子を探しているのか、その理由へ近づいていく。
- **両方を集めきったとき**、断片が一枚の絵のように噛み合い、**物語の全容が立ち上がる**（＝パズルが完成する）。通常色だけでも、レア色だけでも、全体は見えない。
- 狙って集めるというより、ゆっくり拾い続けた人にだけ静かに届く（スロータイムを保つ。急かす通知は出さない）。
- 図書室に断片が溜まり、欠けが埋まっていく。すべて揃うと通しで読める。

### 4.2 出現条件（二系統の収集マイルストーン）

確率ではなく、**シーグラスの累計収集数がしきい値に達したら**小瓶を出す。系統を2つ持ち、どちらも決定論的（リロード・再描画で揺れない）。

```
// 日常の層（通常色）
SURFACE_COLORS   = ["水色","緑","白", ...]   // 通常色プール（要確定）
SURFACE_COUNT    = 5                          // 通常色を累計この数集めるごとに「日常」断片1つ（要調整）

// 核心の層（レア色＝時化限定色, 機能②）
CORE_COLORS      = ["青","赤", ...]           // レア色プール（要確定・機能②のstorm色と一致）
CORE_COUNT       = 3                          // レア色を累計この数集めるごとに「核心」断片1つ（要調整・小さめ）
```

- **累計方式（非消費）**：各系統ごとに、対象色群を規定数集めるごとに、その系統の**次の未取得断片**が1つ届く。集めたガラスは**消えない**（棚に残る。permanence 厳守）。
- **カウントは既存の収集データから導出**：個体の色を数え、`surfaceCount = Σ(通常色)`, `coreCount = Σ(レア色)`。専用カウンタは持たない。
- **判定（各系統独立・決定論）**：
  ```
  surfaceEligible = floor(surfaceCount / SURFACE_COUNT)
  coreEligible    = floor(coreCount    / CORE_COUNT)
  surfacePending  = surfaceEligible - surfaceTriggered
  corePending     = coreEligible    - coreTriggered
  ```
  `*Pending >= 1` なら、その系統の断片を持つ小瓶を1本アーム（発生予約）。溜まっても**1本ずつ**届ける。
- **両系統が必要**：レア色（核心）は入手が遅い＝物語の芯はゆっくり明かされる。通常色（日常）はそれを取り巻く枠を先に埋める。**両系統を完走して初めて全容が揃う**（§4.3）。
- **しきい値の伸び**：既定は各系統で固定。完結が早すぎるなら核心側だけ後半で微増も可（任意・要調整）。
- レア色プール `CORE_COLORS` は**機能②の時化限定色と一致させる**（機能②→③の連結）。よって核心の層は**機能②が前提**。

### 4.2.1 出現のタイミングと場所

- しきい値を跨いだら**即座には出さず、次の 0:00 境界（朝）にひとつの浜へ流れ着く**。漂着（機能②）と同じ"朝の贈りもの"のリズムに揃える。
- 出現する浜：**マイルストーンを跨いだ対象色ガラスを拾った浜**（`foundBeach`）を既定。汀線寄りに配置。
- **帰還挨拶（機能②の 3.6 と共通フック）に接続**できる。核心の断片が届く朝は、挨拶の気配を少しだけ変えてもよい（例：文面の余白）。← 任意。
- 安全弁：**同日に出す小瓶は1本まで**。両系統が同時に跨いだ場合は**核心を優先**し、日常は翌朝へ繰り越す。

### 4.3 断片の並びと全容（パズル）

- 断片は2つのプール（`surface` / `core`）を持ち、各プール内は**順送り**（次の未取得断片を渡す）。
- 各断片は**最終的な通し読みの位置 `order`** を別に持つ。日常と核心が交互・入れ子になって、揃うと一篇（＝物語の全容）として噛み合う。**個々の断片は物語を暗に示すだけで、直接は語らない。**
- **完成条件**：`surface` と `core` の**両方を集めきる**と、図書室の「全容」ビューが解ける。ここで初めて主人公が硝子を探していた理由と物語の全体像が像を結ぶ。
- 片方だけでは完成しない（枠だけ／芯だけでは絵にならない、というパズルの体験）。
- 文面・順序・全容は §4.4 と別ドキュメント（物語設計）で確定する。

### 4.4 文面（⚠️ 著作権）

- **既存の詩・歌詞・書籍からの引用は不可。** Claude Code は既存作品のテキストをコード内に貼り付けないこと。
- 断片は**この作品のために書き下ろした完全オリジナル**のみ。**実在地名を含めない。**
- 文面・二層の役割・順序・全容は **`docs/bottle-narrative.md`（物語設計）** で確定する。本仕様はデータ構造のみ規定する。
- データ構造（プレースホルダ。テキストは物語設計ドキュメントの原稿に差し替える）：

```json
"bottleMessages": {
  "surface": [
    { "id": "s01", "order": 1, "text": "（日常の断片・書き下ろし）" },
    { "id": "s02", "order": 3, "text": "（日常の断片・書き下ろし）" }
  ],
  "core": [
    { "id": "c01", "order": 2, "text": "（核心の断片・書き下ろし）" },
    { "id": "c02", "order": 6, "text": "（核心の断片・書き下ろし）" }
  ]
}
```
> `order` は全容（通し読み）での位置。`surface`/`core` を横断して1本の物語に編み直す。

### 4.5 描画と採取

- 浜に**小瓶の固有オブジェクト**（シード生成のガラス瓶シェイプ）を1点描く。汀線寄りに置くと漂着感が出る。
- 採取すると静かなモーダルで断片を1つ提示（内密な発見の質感）。派手な演出は避ける。

### 4.6 図書室統合

- 既存の二層ライブラリ（種カタログ／棚）に**「断片」ビュー**を追加。
- 収集済み断片を**日常／核心の二層**で表示し、**未取得はプレースホルダ（欠けたピース）**として枠だけ見せる → パズルが埋まっていく感覚。ただし控えめに。
- **両層を完走すると「全容」が解ける**：`order` に沿って断片を編み直し、一篇として通し表示する。ここが物語の到達点。

### 4.7 保存スキーマ（例）

```json
"bottles": {
  "surfaceTriggered": 2,        // 日常の層でこれまでに発生させた断片数
  "coreTriggered": 1,           // 核心の層でこれまでに発生させた断片数
  "armed": [                    // 発生予約済みだが未出現の小瓶（通常0〜1件）
    { "pool": "core", "foundBeach": "<beachId>", "armedDate": "2026-07-18" }
  ],
  "collected": [
    { "pool": "surface", "id": "s01", "order": 1, "foundBeach": "<beachId>", "dateKey": "2026-07-12" },
    { "pool": "core",    "id": "c01", "order": 2, "foundBeach": "<beachId>", "dateKey": "2026-07-15" }
  ],
  "storyComplete": false        // surface/core 両プールを集めきると true → 全容が解ける
}
```
- `surfaceTriggered` / `coreTriggered` としきい値判定（§4.2）で、同じマイルストーンから二重発生しないことを保証する。
- `surfaceCount` / `coreCount`（対象色の累計）は既存の収集データから毎回導出し、専用フィールドは持たない。
- `storyComplete` は両プールの全断片が `collected` に揃った時点で立てる。

### 4.8 受け入れ基準

- [ ] 通常色の累計が `SURFACE_COUNT` に達するごとに「日常」断片が1つ届く。
- [ ] レア色の累計が `CORE_COUNT` に達するごとに「核心」断片が1つ届く。
- [ ] 各系統とも同じマイルストーンから二重に出ない（`surfaceTriggered`/`coreTriggered` で管理）。リロードで再発生しない。
- [ ] マイルストーンを跨いだガラスは消えず、棚に残る（非消費）。
- [ ] 小瓶は次の 0:00 に対象色を拾った浜へ流れ着き、同日に出るのは1本まで（同時到達は核心優先）。
- [ ] 図書室で日常／核心の二層と欠けたピースが見え、**両層を完走すると「全容」が通しで読める**。
- [ ] コード内に外部著作物のテキストが一切含まれない。

## 5. 横断的な留意点

- **後方互換**：既存セーブに新フィールドが無くても壊れないこと。読み込み時に未定義はデフォルト補完。
- **決定論の一貫性**：潮・漂着・小瓶の"今日"はすべて §1.2 の同一日境界に揃える。
- **リセット厳禁**：位置は相対座標(0..1)で保存し、シードと共に永続化。描画は `beach.id` 依存に留め、`beach` オブジェクト再生成を巻き込まない。
- **パフォーマンス**：潮位は毎フレーム再計算しない（1分 tick）。過去天気は当日キャッシュ。
- **世界観**：数値の生表示を避け、座標・地名を露出しない。
